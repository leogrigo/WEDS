Technical Architecture Reference

WEDS Swarm Communication Architecture

A deep-dive into the gateway-orchestrated star topology, alert propagation mechanics, LoRa packet lifecycle, and node duty-cycle state machine of the Wildfire Early Detection System.

Network Topology

WEDS uses a centrally-orchestrated Star topology, where a single gateway acts as the hub for all inter-node communication. Nodes do not communicate directly with each other; all coordination flows through the gateway.

📡

Gateway (Hub)

Heltec ESP32 LoRa V3. Runs continuously, listening for incoming NODE_STATUS and NODE_ALERT packets. Maintains the WedsGatewayRegistry — a live in-memory database of all known nodes. Hosts a REST/WebSocket API for the dashboard. NTP-synced for authoritative timestamps.

🌿

Sensor Nodes (Spokes)

Heltec ESP32 LoRa V3 + BME688. Execute a linear wake → sense → infer → transmit → sleep cycle. Deep-sleep between cycles to conserve battery. Nodes listen for ALERT_MODE_ENABLE commands from the gateway during their post-TX RX window.

🗺️

Geospatial Neighbor Map

The gateway stores GPS coordinates per node (loaded from LittleFS config). When an alert fires, findNeighbors() uses the Haversine formula to identify all nodes within WEDS_NEIGHBOR_RADIUS_M metres of the alerting node and queues ALERT_MODE_ENABLE commands for each.

ℹ️ There is no node-to-node LoRa link. The "swarm" effect is achieved by the gateway acting as a coordinator: it observes an alert from Node A and proactively commands Nodes B and C to enter heightened-alert duty cycles. Nodes are unaware of each other's existence.

Alert Propagation — The Swarm Effect

When any node crosses an alert threshold (anomaly or TinyML risk score), the system orchestrates a cascade that compresses the duty cycles of all geographically neighbouring nodes. Here is the precise step-by-step flow:

1

Node A — TinyML Threshold Breach

Node A's WedsAnomalyDetector or WedsRiskScoreCalculator returns a result where detection_state == WEDS_DETECTION_ALERT (risk score > 0.75 or anomaly score beyond 3σ EMA baseline). The isAlertPayload() helper in main.cpp routes the uplink to nodeComm.sendAlert() instead of sendStatus().

2

Node A — Reliable Alert Transmission

sendAlert() builds a NODE_ALERT packet (msg_type 0x02) with ack_required = WEDS_ACK_REQUIRED. It enters a retry loop (up to WEDS_NODE_ALERT_MAX_RETRIES = 3 attempts, with WEDS_NODE_RETRY_BACKOFF_MS = 300 ms between retries). Each attempt calls waitForAck() with a WEDS_NODE_ACK_TIMEOUT_MS = 1500 ms window.

3

Gateway — Packet Reception & Registry Update

WedsGatewayComm::poll() receives the packet via radio.receive(). handleReceivedPacket() deserializes it and validates the destination (dst_node_id == WEDS_GATEWAY_ID == 0) and magic byte (0xA7). isDuplicateReliablePacket() checks the last sequence ID and msg_type stored in the node's registry record to prevent double-processing on retry. If new, updateNodeStatus() persists the telemetry and updates the event streak log.

4

Gateway — ACK & Neighbor Discovery

The gateway immediately sends an ACK packet back to Node A's node_id. Because the packet is a NODE_ALERT, handleNodeStatusPacket() then calls registry_->createAlertCommandsForNeighbors(source_node_id, status). This function checks isAlertStatus() and runs findNeighbors() using the Haversine formula against all nodes with known GPS coordinates within WEDS_NEIGHBOR_RADIUS_M.

5

Gateway — Queuing Pending Commands

For each neighbor (e.g., Node B, Node C), setPendingAlertCommand() stores a WedsAlertModeEnablePayload in the neighbor's registry record with: alert_source_node_id = Node A's ID, duration_sec = WEDS_ALERT_MODE_DURATION_SEC, sampling_interval_sec = WEDS_ALERT_MODE_SAMPLING_INTERVAL_SEC. The command is held pending until that node's next uplink.

6

Gateway — Opportunistic Command Delivery

After every received packet (status or alert), deliverPendingCommandIfAny(node_id) is called for the sender. If the sender has a pending command, sendAlertModeEnableReliable() immediately attempts delivery: it transmits an ALERT_MODE_ENABLE packet (msg_type 0x10, with ack_required) and waits up to WEDS_GATEWAY_ACK_TIMEOUT_MS for the target node's ACK. On success, clearPendingAlertCommand() removes the pending entry.

7

Node B / C — Alert Mode Activation

During the RX window that follows Node B's next transmission (a 250 ms polling loop in main.cpp), nodeComm.pollAlertModeEnable() receives the ALERT_MODE_ENABLE packet. After validation (correct dst_node_id, src == WEDS_GATEWAY_ID, not a duplicate), it sends an ACK and returns the decoded payload. nodeState.applyAlertModeCommand() then sets alert_mode_active_ = true, records alert_source_node_id_, sets the deadline alert_mode_until_ms_, and overrides the sleep duration. Crucially, it also calls sleep_enabled_ = false, which (in the legacy FreeRTOS path) would prevent the radio from sleeping.

8

Node B / C — Alert Mode Expiry

refreshAlertMode() (called each cycle) checks static_cast<int32_t>(millis() - alert_mode_until_ms_) < 0. Using signed subtraction makes this wrap-safe for durations up to ~24.8 days. Once the timer expires: alert_mode_active_ is cleared, alert_source_node_id_ resets to 0, active_sample_interval_ms_ reverts to normal_sample_interval_ms_, and sleep_enabled_ returns to true.

Packet Lifecycle & Wire Format

Every WEDS transmission follows a fixed binary protocol with a 14-byte header, a variable payload (max 64 bytes), and a 16-bit CRC appended after the payload. The magic byte 0xA7 validates that a received frame belongs to WEDS.

Header Layout (14 bytes)

FieldTypeSizeDescription
magicuint81 BAlways 0xA7 — WEDS frame identifier
msg_typeuint81 B0x01 NODE_STATUS · 0x02 NODE_ALERT · 0x03 ACK · 0x10 ALERT_MODE_ENABLE
ack_requireduint81 B1 = reliable delivery required (NODE_ALERT, ALERT_MODE_ENABLE), 0 = fire-and-forget
src_node_iduint324 BSender MAC-derived ID. Gateway always uses ID 0x00000000
dst_node_iduint324 BRecipient ID. Nodes target gateway ID 0; gateway addresses individual nodes by their MAC ID
sequence_iduint162 BMonotonically incrementing per-sender counter. Used for duplicate detection
payload_lenuint81 BByte length of the payload that follows the header

Payload Structures

Packet TypePayload StructKey FieldsDirection
NODE_STATUS WedsNodeStatusPayload temperature, humidity, pressure, gas_resistance, battery_level, anomaly_state, anomaly_score, risk_state, risk_score Node → Gateway
NODE_ALERT WedsNodeStatusPayload Same structure as STATUS, but ack_required=1 and retried up to 3× Node → Gateway
ACK WedsAckPayload acked_sequence_id, acked_msg_type, status_code (0=OK) Bidirectional
ALERT_MODE_ENABLE WedsAlertModeEnablePayload alert_source_node_id (triggering node), duration_sec, sampling_interval_sec Gateway → Node
🔑 alert_source_node_id — This field inside WedsAlertModeEnablePayload is the propagation link. It carries the node ID of the originally alerting node into every neighbouring node's state (alert_source_node_id_ in WedsNodeState). This is how a node knows which peer triggered its elevated duty cycle, enabling the gateway dashboard to trace alert propagation chains.

Duplicate Suppression

Two independent deduplication systems prevent double-processing of retried reliable packets:

  • Gateway side: WedsGatewayRegistry::isDuplicateReliablePacket() compares the incoming (sequence_id, msg_type) pair against the stored last_sequence_id and last_msg_type for that node. On match, the ACK is re-sent but the registry is not updated again.
  • Node side: WedsNodeComm::isDuplicateGatewayCommand() compares the incoming packet's (src_node_id==gateway, sequence_id, msg_type) against the last processed gateway command. On match, a re-ACK is sent but applyAlertModeCommand() is never called twice.

Timing & Duty Cycle Configuration

The node's sleep duration is determined by two independent mechanisms that can overlap:

🧠

Dynamic Duty Cycle (TinyML-driven)

After every successful cycle, sleepDurationSec(risk.score) maps the model output to a sleep target:

Risk BandThresholdSleep
Low< 35 %1800 s
Medium35–65 %600 s
High≥ 65 %120 s
📢

Swarm Alert Mode (Gateway-driven)

When an ALERT_MODE_ENABLE command is received, the gateway dictates both the period and the duration:

  • duration_sec — How long the node stays in alert mode (set from WEDS_ALERT_MODE_DURATION_SEC)
  • sampling_interval_sec — Replaces the normal sleep period (set from WEDS_ALERT_MODE_SAMPLING_INTERVAL_SEC)
⏱️

RX Window Timing

After every transmission, the node keeps its radio active for a minimum WEDS_NODE_MIN_RX_WINDOW_MS = 250 ms polling window. The window polls every WEDS_NODE_RX_POLL_CHUNK_MS = 50 ms. This is the only period when a node can receive a gateway command. There is no persistent listen mode — commands are exclusively opportunistic.

⚠️ Command Delivery Latency: Because the gateway only delivers pending commands when a node next checks in, the worst-case command delivery latency equals the node's current sleep duration (up to 1800 s = 30 min in low-risk mode). Neighbours with low risk scores may take up to 30 minutes to receive their ALERT_MODE_ENABLE command after a nearby alert fires.
ℹ️ TFLite Inference Awake-Time Budget: TensorFlow Lite Micro's Invoke() is the dominant contributor to node awake time. On first boot (cold kernel init), inference can take 3–8 seconds. The firmware feeds esp_task_wdt_reset() before and after Invoke() to prevent a TWDT panic, and the loopTask stack has been increased to 24 KB (via ARDUINO_LOOP_STACK_SIZE=24576 in platformio.ini) to prevent stack overflow. The additional awake time is automatically absorbed by the virtual timestamp advance: rtc_virtual_timestamp += awake_sec + sleep_sec, so the daily bucket ring buffer stays coherent regardless of inference duration.
