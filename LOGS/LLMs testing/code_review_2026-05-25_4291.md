**Date:** 2026-05-25
**Review ID:** 4291
**Investigator:** Antigravity (Claude Sonnet)
**Scope:** TinyML ``Invoke()`` watchdog reset / crash diagnosis
**Environments tested:** ``WEDS_SIM_MODE_NORMAL``, ``WEDS_SIM_MODE_FIRE_EVENT``, ``WEDS_SIM_MODE_SENSOR_FAULT``

---

## Executive Summary

The ``interpreter_->Invoke()`` call inside ``WedsRiskScoreCalculator::update()`` causes a hard reset on the ESP32S3 (Heltec WiFi LoRa V3). There are **two compounding root causes**:

1. **Primary (Runtime crash): Insufficient Arduino task stack for TFLite Micro** — The firmware runs entirely in the ESP32 Arduino framework``s ``loopTask``. The default stack for this task is 8 KB, and the ``WEDS_NODE_CYCLE_TASK_STACK_BYTES = 8192`` constant does not actually alter this task``s stack in the deep-sleep pipeline (all execution happens in ``setup()``, not a user-created FreeRTOS task). TFLite Micro``s ``Invoke()`` uses deeply nested kernel dispatch tables that push 12-20 KB of stack frames in one shot, immediately overflowing the 8 KB limit. The result is a stack smash -> corrupt return address -> panic reset.

2. **Secondary (Watchdog starvation): Missing TWDT reset around ``Invoke()``** — The ESP32 Arduino framework enables the Task Watchdog Timer (TWDT) with a default timeout of 3 seconds. On the very first inference call, TFLite Micro initializes internal state while executing, and the total wall-clock time for ``Invoke()`` can exceed 3 seconds. This starves the TWDT on Core 1``s idle task and triggers a ``TWDT: Task loopTask did not reset the WDT`` panic.

## Root Cause Analysis

### Cause 1 — Stack Overflow in loopTask (PRIMARY)

**File:** code/include/WedsNodeConfig.h (line 69)
**File:** code/platformio.ini

BEFORE (line 69) — dangerously small:
  static constexpr uint32_t WEDS_NODE_CYCLE_TASK_STACK_BYTES = 8192;

The firmware''s entire execution path lives in setup() -> deep-sleep architecture (no loop() body). The Arduino framework runs setup() and loop() inside a single FreeRTOS task called loopTask. Its stack size is set by the ARDUINO_LOOP_STACK_SIZE preprocessor macro in platformio.ini.

For the [env:node] environment, **this macro was not set**, so it inherited the platform default of **8,192 bytes**. The TFLite Micro AllOpsResolver + MicroInterpreter + Invoke() call chain requires approximately **18-22 KB** of stack. Overflowing 8 KB causes the stack to corrupt adjacent memory (typically the FreeRTOS TCB), producing a crash indistinguishable from a watchdog reset.

**Critical evidence:** [env:gateway] in platformio.ini line 34 DOES set -D ARDUINO_LOOP_STACK_SIZE=24576. The team already knew this fix was needed for complex tasks, but the node environment was overlooked.

### Cause 2 — TWDT Starvation During Invoke() (SECONDARY)

**File:** code/lib/weds_risk_score/WedsRiskScore.cpp (line 102)

BEFORE:
  if (interpreter_->Invoke() != kTfLiteOk) return result;

The ESP32 Arduino framework registers loopTask with the Task Watchdog Timer (TWDT). The default TWDT timeout is 3 seconds. On the first boot cycle, TFLite Micro''s Invoke() triggers cold initialization of the op kernels, which can take 3-8 seconds on an ESP32S3 at 240 MHz. Even after the stack fix, TWDT starvation will cause a panic reset.

WEDS_SIM_MODE_SENSOR_FAULT is the most dangerous scenario: gas_resistance is frozen at the fault value, causing the model to repeatedly see the same out-of-distribution input, potentially extending Invoke() execution time.

### Cause 3 — NaN propagation risk (TERTIARY, latent)

**File:** code/src/node/main.cpp (line 115)

  if (!sample.valid || isnan(sample.temperature) || isnan(sample.humidity)) {

pressure and gas_resistance are NOT checked. A future sensor driver change producing NaN gas resistance would bypass the guard and reach Invoke(), feeding the model a NaN-quantized feature, causing undefined behavior inside TFLite Micro.

## Exact Files and Lines Changed

### Fix 1 — platformio.ini (PRIMARY FIX)

DIFF:
+    -D ARDUINO_LOOP_STACK_SIZE=24576

This forces the Arduino framework to allocate a 24 KB loopTask stack — sufficient headroom for TFLite Micro''s Invoke() call chain (~18-22 KB) plus the firmware''s own call frames.

### Fix 2 — WedsNodeConfig.h (DOCUMENTATION + CONFIG FIX)

DIFF:
- static constexpr uint32_t WEDS_NODE_CYCLE_TASK_STACK_BYTES = 8192;
+ // Increased from 8192 to 24576 — see full comment in file
+ static constexpr uint32_t WEDS_NODE_CYCLE_TASK_STACK_BYTES = 24576;

### Fix 3 — WedsRiskScore.cpp (WATCHDOG FIX)

DIFF:
+#include <esp_task_wdt.h>
 ...
+    esp_task_wdt_reset();
     if (interpreter_->Invoke() != kTfLiteOk) return result;
+    esp_task_wdt_reset();
