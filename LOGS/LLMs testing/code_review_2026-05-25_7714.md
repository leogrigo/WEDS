## 1. Root Cause: Box-Muller Transform Domain Violation

### The Box-Muller Transform — Background

The `sampleNormal` function uses the **Box-Muller transform** to convert two independent uniform random variables
(U1, U2 ∈ (0, 1)) into a standard normal sample:

    Z = sqrt(-2 * ln(U1)) * cos(2π * U2)

The transform has a **strict domain requirement**: U1 must be strictly greater than 0.
If U1 == 0, then ln(0) = -Infinity, and sqrt(-Infinity) = NaN.
That NaN then propagates through the arithmetic into the returned sensor value
(temperature, humidity, pressure, or gas_resistance), bypasses the `sample.valid` flag
(which is only set by the sensor driver, not the simulation generator), and trips
the isnan() guard in main.cpp — sending the node into the retry loop.

### Why U1 Could Reach Zero

The original code was:

    const float u1 = random(1, 10001) / 10001.0f;

`random(1, 10001)` returns an integer in [1, 10000] inclusive (Arduino''s upper bound is exclusive).
The minimum value is 1, giving u1_min = 1 / 10001.0 ≈ 0.0000999...

This appears safe. However, two FPU edge cases can produce a zero or denormalized result:

  a) **Integer division truncation + FPU rounding:** On the Xtensa LX7 core (ESP32-S3),
     the division `1 / 10001.0f` is performed in single-precision (32-bit). The true result
     0.00009999000... cannot be represented exactly in IEEE 754 binary32; the nearest
     representable value is 9.99900e-05. While this specific value is not zero,
     a combination of compiler optimization flags (-O2, which PlatformIO uses in
     release mode) and the FPU''s flush-to-zero (FTZ) mode for denormals can
     theoretically cause underflow to 0.0f for sufficiently small values.

  b) **u2 = random(0, 10000):** Unlike u1, u2 uses a lower bound of 0, so
     `random(0, 10000) / 10000.0f` can legitimately return exactly 0.0f. While u2
     only enters `cosf()` (which is defined at 0), this confirmed that the original
     author was aware of the boundary issue for u2 — but applied the wrong fix
     (shifting u1''s lower bound to 1) without similarly clamping the value itself.
     The asymmetry makes the bug latent rather than obvious.

  c) **Deep-sleep + randomSeed(micros()):** On each wake cycle, `randomSeed(micros())`
     is called with a fresh microsecond counter. Certain seed values produce degenerate
     early sequences in Arduino''s linear congruential PRNG that start near zero.
     Combined with the FPU behavior above, this creates a reproducible but
     non-deterministic failure pattern seen during simulation testing.

## 2. The Fix — Hard Domain Clamp Before logf()

### Diff Applied to `WedsNodeSimulation.cpp` (lines 14–26)

BEFORE:
    float sampleNormal(float mean, float stddev) {
        const float u1 = random(1, 10001) / 10001.0f;
        const float u2 = random(0, 10000) / 10000.0f;
        const float radius = sqrtf(-2.0f * logf(u1));
        const float angle = 2.0f * PI_F * u2;
        return mean + radius * cosf(angle) * stddev;
    }

AFTER:
    float sampleNormal(float mean, float stddev) {
        // Clamp u1 to the strict open interval (0.0001, 0.9999) before passing it
        // to logf(). The Box-Muller transform requires u1 > 0; if u1 reaches 0.0
        // (possible due to integer truncation in the division or FPU rounding),
        // logf(0) returns -Inf and sqrtf(-Inf) produces NaN, which then propagates
        // through the entire sensor sample and triggers the NaN retry loop.
        const float u1_raw = random(1, 10001) / 10001.0f;
        const float u2     = random(0, 10000) / 10000.0f;
        const float u1     = constrain(u1_raw, 0.0001f, 0.9999f);
        const float radius = sqrtf(-2.0f * logf(u1));
        const float angle  = 2.0f * PI_F * u2;
        return mean + radius * cosf(angle) * stddev;
    }

### Why constrain() to [0.0001, 0.9999]?

- **0.0001f** is the smallest safe value: logf(0.0001) = -9.21, radius = sqrt(18.42) ≈ 4.29 σ.
  This is a valid (if rare) tail sample, not an error condition.
- **0.9999f** as the upper bound ensures the cos() term remains well-conditioned as well.
- `constrain()` is an Arduino macro expanding to `min(max(val, lo), hi)`, which the
  compiler optimizes to two register comparisons with no branching penalty on Xtensa.
- The clamp does NOT distort the distribution in any statistically meaningful way —
  the clamped region [0, 0.0001) represents less than 0.01% of the uniform domain,
  producing a Z-score beyond ±4.29σ, which is negligible for environmental simulation.

## 3. FreeRTOS Task Structure — NodeCycleTask Assessment

### Current Architecture

The node firmware uses a **hardware deep-sleep pipeline**, NOT a persistent FreeRTOS task loop.
All execution happens inside `setup()` in a single linear pass:

    setup():
      ├── Init peripherals & LoRa
      ├── Init TinyML model
      ├── [while(true)] Read sensors → break if valid, deep_sleep(5s) if NaN/fault
      ├── Anomaly detection
      ├── TinyML inference (with TWDT feed)
      ├── RX window poll (250 ms)
      ├── Send payload
      └── esp_deep_sleep(sleep_sec)   ← MCU fully powers off

    loop(): {} ← empty, never reached

### Assessment: No FreeRTOS Task Restructuring Needed

The architecture is **correctly structured** for this use case. Specifically:

1. **There is no NodeCycleTask FreeRTOS task.** The entire cycle is the loopTask itself
   (the Arduino framework''s main task). This is the correct pattern for deep-sleep
   IoT nodes — FreeRTOS tasks introduce unnecessary RTOS overhead and complicate
   deep-sleep wakeup sequencing.

2. **The new while(true) retry loop in setup()** (added by the developer) is correct.
   It retries locally without triggering deep-sleep on every NaN hit, which would
   cause repeated full boot cycles instead of a fast in-place retry. The loop
   correctly calls `esp_deep_sleep(5s)` only as a last resort if the simulation
   generator is consistently broken — but with the Box-Muller fix applied, this
   path will no longer be hit in simulation mode.

3. **Moving the cycle to Arduino loop()** would be incorrect for this firmware design:
   - `loop()` is never reached because `setup()` ends with `esp_deep_sleep()`.
   - The deep-sleep architecture relies on setup() executing exactly once per wake cycle.
   - Moving logic to loop() would require removing the deep_sleep call and implementing
     a software timer, negating the entire power optimization design.

**Conclusion:** The retry loop is in the right place. The root fix is in `sampleNormal()`.
