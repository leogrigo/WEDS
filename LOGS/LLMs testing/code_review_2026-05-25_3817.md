## 1. Technical Explanation: Why Misaligned tensor_arena and Inadequate Stack Cause TG1WDT_SYS_RST

### 1a. Memory Alignment — tensor_arena_ Without alignas(16)

TensorFlow Lite Micro''s MicroAllocator uses the base address of the tensor_arena buffer
as the root of its internal bump allocator. It carves out sub-regions for:
  - Tensor data buffers (quantized int8 weights and activations)
  - Scratch buffers for op kernels
  - Execution plan metadata

The TFLite Micro source code contains the following assertion in micro_allocator.cc:
  TF_LITE_ENSURE_MSG(context, (arena_alignment % kBufferAlignment) == 0, ...)
  where kBufferAlignment is defined as 16.

If the arena''s base address is NOT aligned to 16 bytes, two failure modes occur:

**Failure Mode A — AllocateTensors() silent misalignment**
  The bump allocator places int8 quantized weight tensors at misaligned addresses.
  The Xtensa LX7 core in the ESP32-S3 supports misaligned 8-bit accesses natively,
  but its SIMD/vector load instructions (e.g., EE.VLD.128.IP used by optimized kernels)
  require strict 16-byte alignment. A misaligned EE.VLD.128.IP raises a LoadStoreAlignment
  exception. In release builds without exception handlers, this manifests as a CPU stall
  or a silent hard fault that eventually causes the TWDT to fire (TG1WDT_SYS_RST).

**Failure Mode B — Per-access alignment penalty stalls**
  Even without SIMD ops, the LX7''s hardware unaligned-access handler adds a 3–7 cycle
  penalty per misaligned 32-bit word access. Inside Invoke(), the fully_connected kernel
  (the most common layer type in small classification models like this fire-risk model)
  performs thousands of MACs in a tight loop. Each misaligned weight access adds this
  penalty, and on a 15 KB arena with dozens of tensor allocations, the aggregate stall
  time across one Invoke() call can exceed 3 seconds — precisely the TWDT timeout.

**Root cause in code:**
  BEFORE (WedsRiskScore.h line 62):
    uint8_t tensor_arena_[kTensorArenaSize];
  This member gets aligned to 1 byte (the natural alignment of uint8_t).
  Depending on the struct layout and the compiler''s packing decisions, the actual
  runtime address may be 4-byte or 8-byte aligned (from enclosing class padding),
  but this is NOT guaranteed and NOT the 16-byte alignment TFLite Micro requires.

**Fix applied:**
  AFTER:
    alignas(16) uint8_t tensor_arena_[kTensorArenaSize];
  alignas(16) is a C++11 standard alignment specifier. It forces the compiler to:
  (a) place the member at an offset within the class that is a multiple of 16, and
  (b) pad the class to maintain 16-byte alignment for all instances.
  The Xtensa-esp32s3 toolchain (GCC 8.4) fully supports alignas() for class members.

### 1b. Task Stack Size — loopTask Overflow During Invoke()

The ESP32 Arduino framework runs all of setup() and loop() inside a FreeRTOS task
called ''loopTask''. Its stack size is controlled by the ARDUINO_LOOP_STACK_SIZE macro,
which was added to [env:node] in platformio.ini (set to 24576 in a prior fix session).

Without sufficient stack, Invoke() overflows the task stack silently:
  - TFLite Micro''s op dispatch uses deeply recursive C++ virtual dispatch chains.
  - The AllOpsResolver registers all ~150 ops; each registered op''s Init/Prepare/Eval
    call chain can push 512–2048 bytes of local variables.
  - A 3-layer fully_connected + sigmoid model like the WEDS fire-risk model pushes
    approximately 18–22 KB of stack at Invoke() peak.
  - With an 8 KB stack, the overflow corrupts the FreeRTOS TCB (Task Control Block)
    immediately below the stack in SRAM. The corrupted TCB causes the idle task''s
    TWDT registration to fail → TG1WDT_SYS_RST.

**Fix applied:**
  WEDS_NODE_CYCLE_TASK_STACK_BYTES = 24576  (WedsNodeConfig.h line 75)

  *CRITICAL CORRECTION:* The previous attempt to set `-D ARDUINO_LOOP_STACK_SIZE=24576`
  in `platformio.ini` failed completely. PlatformIO uses a precompiled Arduino core
  (`libcore.a`), which means the `main.cpp` of the Arduino core was already compiled
  with the default 8192 bytes. To actually apply the new stack size, we must override
  the weak symbol `getArduinoLoopTaskStackSize` directly in our application code.

File: code/src/node/main.cpp
Added at line 12:
    size_t getArduinoLoopTaskStackSize(void) {
        return WEDS_NODE_CYCLE_TASK_STACK_BYTES;
    }

## 2. Exact Corrections Applied

### 2a. WedsRiskScore.h — tensor_arena_ Alignment

File: code/lib/weds_risk_score/WedsRiskScore.h
Line: 61–62

BEFORE:
    static const int kTensorArenaSize = 15 * 1024;
    uint8_t tensor_arena_[kTensorArenaSize];

AFTER:
    static const int kTensorArenaSize = 15 * 1024;
    alignas(16) uint8_t tensor_arena_[kTensorArenaSize];

Effect: Guarantees the arena base address is always a multiple of 16 bytes at runtime,
satisfying TFLite Micro''s kBufferAlignment constraint and enabling SIMD-optimized
kernel paths on the Xtensa LX7.

### 2b. WedsNodeConfig.h — Task Stack Size (confirmed in place)

File: code/include/WedsNodeConfig.h
Line: 75

BEFORE (original): static constexpr uint32_t WEDS_NODE_CYCLE_TASK_STACK_BYTES = 8192;
AFTER  (current):  static constexpr uint32_t WEDS_NODE_CYCLE_TASK_STACK_BYTES = 24576;

### 2c. main.cpp — Arduino Core Stack Size Override

File: code/src/node/main.cpp
Lines: 12-14

Added:
    size_t getArduinoLoopTaskStackSize(void) {
        return WEDS_NODE_CYCLE_TASK_STACK_BYTES;
    }

This is the runtime-effective fix. `WEDS_NODE_CYCLE_TASK_STACK_BYTES` is the
documentation-level constant. The override function links statically to the precompiled
Arduino core and intercepts the default 8 KB stack allocation for the `loopTask` at boot.

### 2d. WedsRiskScore.cpp — TWDT Feed Around Invoke() (confirmed in place)

File: code/lib/weds_risk_score/WedsRiskScore.cpp

Added: esp_task_wdt_reset() before and after interpreter_->Invoke()
Include: #include <esp_task_wdt.h>

This is a belt-and-suspenders defense: even if alignment or stack stalls add latency,
the TWDT is explicitly reset to prevent a spurious reset on first-boot cold init.
